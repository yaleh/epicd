# Proposal: 全面 epicd 品牌化

> 本文是 proposal（pre-decision 设计稿），**不含 plan**。它推翻 BACK-687/688/691
> 的 Non-Goals 中"保留 backlog 品牌/协议标识"的裁定，将审计中发现的全部残留
> backlog/Backlog.md 字样统一改为 epicd 品牌，项目 URL 统一为
> `https://github.com/yaleh/epicd`。

## 1. 背景

BACK-681/687/688/691 完成了 CLI 二进制名、CLI 调用示例、shell 补全脚本、以及
~63 处用户可见品牌字符串从 "backlog"/"Backlog.md" 到 "epicd" 的改名，但显式把以下
5 类识别为"不变量"排除在外：`MCP_SERVER_NAME` 常量值、`backlog://` MCP URI
scheme、`backlog/` 任务存储目录名、npm optionalDependencies 平台包名、上游归属
URL `github.com/MrLesk/Backlog.md`。

2026-07-14 一次 meta-cc 仓库任务收尾故障排查（`epicd task edit --phase` 在
legacy 任务上必然失败）过程中，进一步发现本机 `~/.claude.json` 的 MCP 注册
`command: "backlog"` 解析到一个 2026-05-07 装的、早于改名的全局
`backlog.md@1.45.0` 包，而非本仓库正在开发的 `epicd@1.48.16`。据此对全仓库做了
一次完整的 backlog/Backlog.md 残留审计，发现：

1. `src/agent-instructions.ts` 会把带 `BACKLOG.MD GUIDELINES` marker 的指南块
   **写入外部消费者仓库**的 `AGENTS.md`/`CLAUDE.md` 等文件——这是持续污染的分发型
   问题，不只是本仓库自身卫生问题。
2. 除已知的 5 类不变量外，还存在若干未被 BACK-691 覆盖的品牌残留（展示层字符串、
   一处已随 optionalDependencies 改名而失效的正则、若干 README 段落）。
3. BACK-699/700 已经把"目录名"这一不变量部分推翻（新项目默认 `.epicd/`，本仓库
   自身已迁移），但 BACK-687/688/691 里关于 `MCP_SERVER_NAME`/`backlog://`/上游
   URL 的条款仍然有效，从未被推翻。

本文的决策：**全部推翻**，包括协议层标识符（`MCP_SERVER_NAME`、`backlog://`、
MCP 工具名），不做过渡期双重兼容，只在文档中写清迁移步骤。

## 2. 范围之外的说明

`src/utils/backlog-directory.ts`、`src/core/init.ts` 中用于**检测历史仓库目录
名**的字面量 `"backlog"`/`".backlog"`（如 `BacklogDirectorySource` 类型、
`backlogDir`/`backlogPath` 变量名及其值）**不在本次改动范围内**。这些是磁盘兼容
性探测逻辑，改动会破坏对存量仓库（仍用 `backlog/`/`.backlog/` 目录）的识别，属于
功能回归而非品牌问题，与 BACK-699/700 已确立的"新项目默认 `.epicd/`、旧仓库永久
兼容"方向保持一致，不受本提案影响。

**审阅补充（同一排除理由适用、原稿未列出的文件，需同步纳入排除范围而非遗漏）**：

  `backlogResolution`/`backlogJson`，以及对遗留 `backlog.json` 文件名的探测逻辑，

  遗漏。

以上补充只是把 Section 2 已确立的排除理由应用到审计中新发现的同类文件，不改变

### 迁移说明（写入 README 或 CHANGELOG，供外部用户参考）

- 已通过 `claude mcp add -s user backlog -- ...` 注册的用户，需要执行
  `claude mcp remove backlog && claude mcp add -s user epicd -- epicd mcp start`
  重新注册（server key 从 `backlog` 变为 `epicd`）。
- 任何硬编码 `backlog://...` 资源 URI 或按名调用 `get_backlog_instructions` 工具
  的外部脚本/集成，需要改为 `epicd://...` 与 `get_epicd_instructions`。
- 仓库内如有旧版 `epicd init` 生成的、带 `<!-- BACKLOG.MD GUIDELINES -->` marker
  的指南块，需要手动删除后重新运行 `epicd init` 以生成新 marker 块；否则新旧两
  块会并存。
- 全局安装的旧 `backlog.md` npm 包（`npm i -g backlog.md`）不再是当前维护对象，
  建议 `npm uninstall -g backlog.md` 并改用 `npm i -g epicd`。

## 6. 风险与未决问题

- **外部消费者破坏性**：Batch 2 的每一项都会立即打破尚未升级的外部
  MCP 客户端（server key、资源 URI、工具名）。本提案选择硬切 + 文档迁移说明，
  不做双重兼容，接受这一权衡（epicd 目前用户规模小、仍处于快速改名收尾阶段）。
- **npm 包 `backlog.md`**：旧包名已发布在 npm registry（`backlog.md@1.48.0`），
  无法删除，只能后续单独发起 deprecate 流程（不在本提案范围）。
- **上游归属**：`github.com/MrLesk/Backlog.md` 的 fork 归属信息（如 LICENSE 中的
  版权声明）不在本提案讨论范围内，只处理正文散文中的品牌/链接引用；LICENSE 本身
  的处理留待单独确认。

### 6.1 审阅发现（2026-07-14 独立审阅，判断类结论）

- **审计完整性结论**：本提案标题/引言称"全仓库审计"，但独立复核（`grep -rni

  `CLI-INSTRUCTIONS.md`、`src/mcp/README.md`、
  `scripts/smoke-parallel-task-locking.sh`、`docs/uml/use-case-model.md`）以及
  `MCP_GUIDE_URL` 这一独立的协议层常量，README.md 内的散文条目数也从"约11处"修正
  为实际 17 处。已在上文表格与 `applies-to` 中补全。这类遗漏说明审计当时可能只
  搜索了应用层源码目录而未覆盖仓库根目录的非 `.ts`/`.md`（`src/`外）文件与
  `.claude-plugin/`；建议实施阶段以 `grep -rni backlog` 的完整输出作为改动清单

- **Batch 2 硬切 vs. 轻量兼容的权衡（已裁决，见下）**：提案在 §6 只论证了"接受
