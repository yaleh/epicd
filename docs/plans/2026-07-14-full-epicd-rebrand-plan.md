# Plan: 全面 epicd 品牌化

> 本文档只做实施规划，不含实现代码。范围与判断依据完全来自
> `docs/proposals/2026-07-14-full-epicd-rebrand.md`（以下简称"提案"），
> 逐条对照请以该文件为准；本文档不重复提案的背景/风险论证，只组织成可执行的
> Phase/Stage/AC。

## 0. 任务粒度判断：单个 Basic 任务，不拆 Epic

按 CLAUDE.md 的 Epic 判定标准（两个条件必须同时成立才拆分）：

- **(a) ≥2 个独立可评审/可合并的交付物？** 否。提案的 Batch 1（展示层文案）、
  Batch 2（协议标识符硬切）、Batch 3（历史任务追加评论）不是三个独立可合并的
  交付物——它们是同一个"品牌化推翻不变量"决策的三个先后步骤，且存在硬依赖：
  - Batch 2 的 `MCP_GUIDE_URL` 正确性依赖 Batch 1 对 README 标题的"保留不变"处理，
    两者必须在同一次评审中一起校验，拆开会让其中一个 PR 单独看不出这条依赖。
  - Batch 1 表格里 `AGENTS.md` 第 100/102 行的 marker 块、`docs/uml/use-case-model.md`
    的工具名引用，都要等 Batch 2 的常量改名落地后才有正确的目标值——文本上属于
    "展示层"，但内容上是 Batch 2 的连带产物，硬拆会产生一个中间态不一致的提交。
  - Batch 3 只是对 5 个历史任务文件追加一条评论（一次 `task_edit --comment` 循环），
    体积和独立性都够不上"可评审交付物"，本质是 Batch 1/2 决策生效后的记录动作。
- **(b) 合并后总规模是否有 ≥1.8-2x 于 2000 行上限的余量（≈3600+ 行）？** 否。
  提案 §3/§4 列出的改动点：Batch 1 约 20 个文件、多数是个位数处的字符串替换
  （README 17 处散文 + 2 处渠道行是最大头，其余文件多为 1-4 处）；Batch 2 是
  7 组标识符改名（含本轮复核新增的 `BACKLOG_CWD` 环境变量名与 README MCP 章节的
  协议字面量），涉及文件数约 15-18 个（含 4 篇 guideline md、5 个 MCP 相关测试
  文件、2 个 `BACKLOG_CWD` 相关测试文件）。按可比历史经验（"单个 API 扩展或
  point fix 历史上在 1000-1500 行"），Batch 1 + Batch 2 的合计预期仍在千行量级，
  明显达不到 3600+ 行门槛，也大概率达不到 2000 行上限本身；本轮复核新增的条目
  只是把文件计数从"不超过 12 个"上修到"约 15-18 个"，未改变这一结论。

**结论**：整个改动组织为**一个 Basic 任务**，内部用 3 个 Phase（对应提案的
Batch 1/2/3）+ 每个 Phase 若干 Stage 来管理可恢复检查点，不创建 Epic 或多个子任务。

## 5. §6.1 未决判断的处理：决定纯硬切，不做轻量兼容分支

提案 §6.1 第二条标记了一个未决判断：是否为 `MCP_SERVER_NAME` 增加一个"检测到
旧 server key `backlog` 时打印 deprecation 提示、仍按 `epicd` 运行"的轻量兼容
分支。

**本计划的决定：不做，维持提案主文的纯硬切。** 理由：

1. 这个提示分支的触发条件依赖外部调用方仍在用字符串 `"backlog"` 作为 server
   key 去调用——但 `MCP_SERVER_NAME` 是被动的常量值，用来生成 `claude mcp add`
   等安装命令的 server key，并不是一个被外部输入解析、可以在运行时"检测到旧
   值"的入口。要实现"检测旧 key"实际上需要在 CLI 参数解析层增加一个新的分支
   （例如识别 `--server-name backlog` 或类似调用），这不是"几行代码的提示"，
   而是新增一个需要独立测试覆盖的兼容路径，与提案"不引入双运行时机制"的初衷
   相悖——即使不算"双注册"，也是双路径。
2. 提案 §6 已经给出"用户规模小、接受硬切"的整体权衡，且 Phase 2 的迁移说明
   （提案 §4 "迁移说明"小节）已经把手工修复步骤写清楚，纯硬切 + 文档迁移说明
   的组合对当前规模是相称的。
3. 若未来发现有真实用户因此受阻，可以在独立的后续任务里补一个 CLI 层的
   deprecation 提示，不需要现在为了一个假设的边际收益扩大本任务范围。

因此本计划**不新增 Stage** 来实现该提示分支；Phase 2 Stage 2.1 按纯硬切执行。
